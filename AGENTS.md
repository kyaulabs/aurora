# AGENTS.md

## Stack

- OS: Linux
- Shell: bash
- Web Server: nginx
- Database: MariaDB (MySQL-compatible)
- Backend: PHP 8.5+
- Frontend: HTML5, CSS3, JavaScript (ES6+), jQuery (only when vanilla JS is insufficient)
- CSS Preprocessor: SCSS compiled via sass (Dart Sass)
- JS Minifier: uglify-js
- Version Control: Git

## Boundaries

- Do not access external APIs or services without explicit permission
- Do not modify files outside the project directory
- Do not execute commands that could harm the system or compromise security
- Do not generate code that violates established coding standards
- NEVER modify cdn/css/*.min.css or cdn/javascript/*.min.js — these are generated by external tools and any changes will be overwritten
- CSS minified files are generated from cdn/sass/*.scss via `sass --style=compressed`
- JavaScript minified files are generated from cdn/js/*.js via `uglifyjs`
- New dependencies must be explicitly noted (packages are audited monthly)
- Environment variables belong in .env.example, NEVER commit .env files
- When a file search tool (glob, grep) returns empty results unexpectedly,
  cross-verify with `ls` or `read` before concluding a file or directory
  does not exist. Tools can return stale/missed results on any platform,
  especially for hidden paths (dotfiles, `.github/`).

## No MVC

- We do not use Model-View-Controller
- PHP is flat procedural / class-based with require/include for dependencies
- Each .php page includes Aurora, outputs HTML directly, and interacts with the
  database via raw SQL or Aurora's SQL handler
- The backend/ directory contains PHP logic (classes, functions, DB helpers)
  that is not web-accessible
- There is no router, no controller classes, no templating engine beyond
  Aurora's header/footer methods

## File Naming Conventions

- PHP (procedural helpers, config files, etc.): snake_case (index.php, config_loader.php)
- PHP (class, interface, or trait): PascalCase (ClassName.php, InterfaceName.php, TraitName.php)
- All other files: snake_case (my_script.js, site_styles.scss)
- Time-stamped files: append `-YYYYMMDDThhmmss` (report-20240722T143000.php)
- Test files: PascalCase suffixed with `Test.php` (UserAuthenticationTest.php)

## Commenting Standards

### RCS-style Header (REQUIRED on every file)

Every source file must begin with an RCS-style identifier. Update the version
and date on every modification.

```text
PHP:  <?php # $KYAULabs: file.php,v 1.0.0 YYYY/MM/DD hh:mm:ss -0700 kyau Exp $
SCSS: // $KYAULabs: file.scss,v 1.0.0 YYYY/MM/DD hh:mm:ss -0700 kyau Exp $
JS:   // $KYAULabs: file.js,v 1.0.0 YYYY/MM/DD hh:mm:ss -0700 kyau Exp $
Bash: # $KYAULabs: file.sh,v 1.0.0 YYYY/MM/DD hh:mm:ss -0700 kyau Exp $
```

### PHPDoc (PSR-5)

- All PHP classes, methods, and functions must have PHPDoc docblocks
- Document parameters, return types, and exceptions

### End-of-File

- All files end with a vim modeline:
  PHP:  `// vim: ft=php sts=4 sw=4 ts=4 et :`
  SCSS: `// vim: ft=scss sts=2 sw=2 ts=2 et :`
  JS:   `// vim: ft=javascript sts=4 sw=4 ts=4 noet :`
- PHP: 4-space indentation (PSR-12)
- SCSS: 2-space indentation (stylelint default)
- JavaScript: tab indentation, tab stops at 4

### No extra comments

- Do not add explanatory comments to code unless explicitly requested

## Testing

- Framework: [Pest PHP v4](https://pestphp.com/) on top of PHPUnit 12
- Directory: `tests/` with subdirectories per category (Unit/, Feature/, Integration/, Browser/)
- File naming: PascalCase (UserAuthenticationTest.php)
- Minimum code coverage: 80% line coverage
- Run all tests: `php vendor/bin/pest --coverage`
- Run parallel: `php vendor/bin/pest --parallel`
- Run browser tests in debug/headed mode: `php vendor/bin/pest --debug`

### Test-Driven Development (TDD) — MANDATORY

**All new code must follow TDD. No exceptions.**

The TDD cycle is:

1. **Red** — Write a failing test that describes the desired behavior before writing any production code.
2. **Green** — Write the minimum production code needed to make the test pass.
3. **Refactor** — Clean up production code and tests, keeping all tests green.

Never write production code without a corresponding failing test already in place.
Never write a test after the fact just to satisfy coverage; tests must drive the implementation.

### What Makes a Good Test

A good test verifies real behavior, not implementation details. When writing tests, always cover:

- **Happy path** — the normal, expected input and output.
- **Boundary conditions** — empty strings, zero, null, maximum/minimum values.
- **Invalid input / error paths** — what happens when the function receives bad data, wrong types, or out-of-range values.
- **Side effects** — database writes, file changes, external calls (mock these).
- **State changes** — ensure the system is in the correct state before AND after the operation.

Do NOT write tests that:

- Only assert `true` or that a method exists (these test nothing meaningful).
- Duplicate another test with a trivially different description.
- Exist solely to push a coverage percentage past a threshold.
- Test internal/private implementation rather than observable behavior.

### Test Structure

Each test file corresponds to one class or module under test. Use `describe()` blocks to
group related scenarios. Use `it()` for individual cases.

```php
<?php # $KYAULabs: user-authentication-test.php,v 1.0.0 YYYY/MM/DD hh:mm:ss -0700 kyau Exp $

describe('UserAuth::login()', function () {
    it('returns a session token on valid credentials', function () {
        // arrange
        $auth = new UserAuth($this->db);
        // act
        $token = $auth->login('user@example.com', 'correct-password');
        // assert
        expect($token)->toBeString()->not->toBeEmpty();
    });

    it('throws InvalidCredentialsException on wrong password', function () {
        $auth = new UserAuth($this->db);
        expect(fn () => $auth->login('user@example.com', 'wrong'))
            ->toThrow(InvalidCredentialsException::class);
    });

    it('throws InvalidCredentialsException on non-existent user', function () {
        $auth = new UserAuth($this->db);
        expect(fn () => $auth->login('nobody@example.com', 'password'))
            ->toThrow(InvalidCredentialsException::class);
    });

    it('rejects an empty email', function () {
        $auth = new UserAuth($this->db);
        expect(fn () => $auth->login('', 'password'))
            ->toThrow(InvalidArgumentException::class);
    });
});
```

Use datasets for testing the same behavior across many inputs:

```php
it('rejects invalid email formats', function (string $email) {
    $auth = new UserAuth($this->db);
    expect(fn () => $auth->login($email, 'password'))
        ->toThrow(InvalidArgumentException::class);
})->with([
    'missing @'     => ['notanemail'],
    'missing domain'=> ['user@'],
    'empty string'  => [''],
]);
```

### Test Categories

- **Unit** (`tests/Unit/`) — Test a single class or function in isolation. Mock all dependencies.
- **Feature** (`tests/Feature/`) — Test a complete feature or user-facing behavior end-to-end
  at the HTTP/PHP level (no real browser). Hit actual database (use transactions, roll back after each test).
- **Integration** (`tests/Integration/`) — Test interactions between two or more real subsystems
  (e.g., your SQL handler + MariaDB schema).
- **Browser** (`tests/Browser/`) — Pest v4 browser tests using the browser plugin + Playwright.
  Reserve for critical UI flows (login, checkout, critical forms). Not for every page.

### Pest v4 Browser Testing Setup

Install the browser plugin and Playwright:

```bash
composer require pestphp/pest-plugin-browser --dev
npm install playwright@latest
npx playwright install
```

Add to `.gitignore`:

```text
tests/Browser/Screenshots/
```

Configure global browser options in `tests/Pest.php`:

```php
pest()->browser()->timeout(10000);
// pest()->browser()->headed(); // uncomment locally for visual debugging
```

Run browser tests in CI (GitHub Actions — add to workflow):

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: lts/*
- name: Install JS dependencies
  run: npm ci
- name: Install Playwright browsers
  run: npx playwright install --with-deps
```

Example browser test skeleton:

```php
it('loads the homepage without JS errors', function () {
    $page = visit('/');
    $page->assertSee('Expected Heading')
         ->assertNoJavaScriptErrors()
         ->assertNoConsoleLogs();
});

it('loads all public pages without smoke', function () {
    $pages = visit(['/', '/about', '/contact']);
    $pages->assertNoSmoke();
});
```

### Retrofitting Tests onto an Existing Project

When adding tests to a project that has none:

1. Run `php vendor/bin/pest --coverage` to see the current baseline (likely 0%).
2. Identify the highest-risk, most-called classes/functions first (auth, DB helpers, billing).
3. For each, read the existing implementation and write tests that describe what it *should* do,
   not just what it *happens* to do. Expose any bugs found in the process.
4. Do NOT chase coverage percentages — cover behavior. Coverage will follow.
5. Once core logic is covered, add Feature tests for critical HTTP paths.
6. Add Browser tests last, only for the most critical UI flows.

### Architecture Testing

Use Pest's arch testing in `tests/Pest.php` to enforce codebase rules automatically:

```php
arch('no debug functions in production code')
    ->expect(['dd', 'dump', 'var_dump', 'print_r'])
    ->not->toBeUsed();

arch('backend classes use strict types')
    ->expect('App')
    ->toUseStrictTypes();
```

#### Responsive / Mobile-First Design

- All styling is developed mobile-first: base styles target the smallest viewport,
  with `min-width` media queries layering on complexity for larger screens
- Sites must work on all devices (320px mobile, 768px tablet, 1024px+ desktop)
- Prefer relative units (`rem`, `%`, `vw`, `vh`, `clamp()`) for layout and spacing;
  avoid fixed `px` values for widths and margins
- Use `clamp()` for fluid typography and spacing that scales with viewport width
- Avoid fixed pixel widths on layout containers; use `max-width` + responsive padding
- Touch targets must be at least 44×44px for mobile accessibility

## Aurora API

Never guess Aurora method signatures — read `aurora/aurora.inc.php` and `aurora/sql.inc.php` before use.

### KYAULabs\Aurora (aurora.inc.php)

Properties set via magic `__set()`: `title`, `description`, `dns` (array), `css` (array), `js` (array), `mjs` (array), `preload` (array).

- `htmlHeader(): bool` — renders template with variables; must be called after all properties are set
- `htmlFooter(): bool` — renders closing `</body></html>` and script tags
- `comment(array $rus, string $script, bool $vim): string` — returns HTML comment with version + render time; call via `echo $site->comment($rus, $_SERVER['SCRIPT_FILENAME'], true)`
- `version(string $script): ?string` — returns version string from RCS header
- `testVariables(): string` — debug helper; reports replaced template variables

### KYAULabs\SQLHandler (sql.inc.php)

Credentials come from `aurora/settings.inc.php` (never hardcode). Constructor: `new KYAULabs\SQLHandler('dbname')`.

- `query(string $sql, array $args = []): PDOStatement|bool` — always use named (`:param`) or anonymous (`?`) parameters; never interpolate user input into SQL
- `setDatabase(string $name): bool` — switch active database
- `$handler->pdo` — direct PDO instance; available if raw PDO access is needed
- `$handler->result` — last PDOStatement result
- Error modes: `SQLHandler::INTERNAL_HANDLING` (default, dies with trace), `SQLHandler::THROW_EXCEPTION`, `SQLHandler::IGNORE_ERRORS`

## Database Conventions

- Tables: `snake_case`, plural (e.g. `users`, `login_attempts`, `users_tokens`)
- Primary key: `id serial` (auto-increment bigint unsigned)
- Foreign keys: named `tablename_col_foreign`, always with `ON DELETE CASCADE ON UPDATE RESTRICT`
- UUIDs stored as `binary(16)`; use `UUID_TO_BIN()` / `BIN_TO_UUID()` helpers
- IP addresses stored as `int(32) unsigned`; use `INET_ATON()` / `INET_NTOA()`
- Timestamps: `timestamp NOT NULL DEFAULT NOW()`; use `FROM_UNIXTIME()` / `UNIX_TIMESTAMP()` when needed
- Charset: `utf8mb4`, collation `utf8mb4_uca1400_ai_ci`; ASCII-only fields (hashes, tokens) use `ascii_bin`
- Engine: `InnoDB`
- All columns require a `COMMENT`
- Indexes: `USING BTREE`; unique constraints via `UNIQUE KEY`

## Security

- All SQL queries must use prepared statements via `SQLHandler::query()` with bound parameters — never interpolate variables into SQL strings
- Never use `eval()`, `exec()`, `shell_exec()`, `system()`, or `passthru()`
- Never echo unsanitized user input; always use `htmlentities()` or `htmlspecialchars()` before output
- Passwords: store only bcrypt hashes (`password_hash($pass, PASSWORD_BCRYPT)`); never log or echo plaintext passwords
- Tokens: use `binary(16)` with `random_bytes(16)`; use `hash_equals()` for comparison
- Sessions: regenerate session ID on privilege change (`session_regenerate_id(true)`)

## Error Handling

- Aurora throws `KYAULabs\AuroraException` — do not suppress with `@` on Aurora calls
- SQLHandler defaults to `INTERNAL_HANDLING` (dies with trace on error); set `$handler->err = SQLHandler::THROW_EXCEPTION` when you need to catch DB errors explicitly
- PHP errors: Aurora sets `display_errors=1` when `$status=true` (dev), `0` when false (prod) — do not override this
- Never use `die()` or `exit()` directly in application code; let Aurora/SQLHandler handle fatal errors
